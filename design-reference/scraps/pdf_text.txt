2. Copy Conventions & Tone
2.1 Brand Voice
• Friendly but professional — like a helpful shopkeeper
• Confident, not apologetic
• Clear over clever — avoid jargon and cutesy phrases
• Respectful of user's time — short sentences, scannable
• Warm but not saccharine
2.2 Do's
• "Order placed!" not "Your order has been successfully placed"
• "Try again" not "Please try again"
• "Delivery photo required" not "Kindly capture the delivery photograph"
• "Address saved" not "Your address has been saved successfully"
2.3 Don'ts
• Avoid 'Please' everywhere — use sparingly for genuine apology
• Avoid 'Kindly' — too formal for mobile
• Avoid 'successfully' — it's noise; if we're showing the message, it succeeded
• Avoid exclamation marks except for genuine celebration (order placed, subscription
created)
• Avoid 'Oops' or similar — feels infantilizing in serious contexts (payment failures)
2.4 Error Message Format
Every error message answers three questions:
• What happened? (in one short phrase)
• Why? (if useful to user)
• What can they do? (action to take)
Examples:
BAD: "Error: Transaction failed. Code 402."GOOD: "Payment failed. Your card was
declined — try another payment method."BAD: "An error occurred. Please try
again."GOOD: "Couldn't place order. Check your connection and retry."BAD: "Invalid
input."GOOD: "Phone number should start with 6, 7, 8, or 9."
2.5 Button Copy
• Action verbs always: "Send OTP" not "OTP"
• Primary CTAs are specific: "Pay ₹845" not "Pay now"
• Destructive actions: red color + clear verb: "Delete" not "Remove" (when permanent)

• Cancel/back buttons: "Cancel" for decisions, "Back" for navigation
• Avoid "Submit" — use the actual action: "Create account", "Place order", "Send invite"
2.6 Empty State Copy
Every empty state has:
• Why it's empty (not accusatory, just factual)
• What user can do about it (CTA)
• Illustration tone: friendly, not sad
Examples:
No orders yet: "No orders yet" "Start by browsing vendors near you" CTA: "Browse
vendors"No addresses saved: "No saved addresses" "Save an address to order faster next
time" CTA: "Add address"
2.7 Confirmation Copy
• Success toasts are short: "Address saved", "Review posted"
• Destructive confirmations spell out consequence: "Delete this address? Active orders
using it will not be affected."
• Irreversible actions include word "permanent": "Account deletion is permanent after 30
days"
2.8 Regional Sensitivity
• Avoid westernisms (no "awesome", "rockstar")
• Amounts in Indian format: ₹1,45,000 (not ₹145,000)
• Dates default to DD/MM/YYYY in display; ISO in backend
• Times in 12-hour with AM/PM
• Address order: flat → building → street → area → city → pincode

3. Design Patterns & Components
3.1 Core Principles
P1 — Thumb-first
• Primary CTAs at bottom; back arrow at top-left
• Critical actions within bottom 40% of screen
• Long forms use progressive disclosure, not scroll-to-bottom
P2 — Transparent
• All fees shown upfront at cart, not surprise at payment
• Delivery time shown honestly (include buffer, don't under-promise)
• Surge multiplier disclosed as distinct line item
P3 — Low literacy friendly
• Every icon paired with label for critical actions
• Numeric keyboards for phone/OTP/amount inputs
• Audio cues for vendor/driver alerts (not silent UI)
• Language switcher top-right, always accessible
P4 — Network-aware
• Offline banner at top when disconnected
• Cached lists still usable offline (greyed out)
• Form drafts saved locally; restore on return
• Photo uploads in background, retry on failure
P5 — Role-aware accent colors
• Customer: blue (#2E75B6)
• Vendor: teal (#17A2B8)
• Driver: orange (#F07820)
• Admin: purple (#7B2CBF)
• Role indicator visible in every top bar

Error / edge cases
• Network completely absent: wait 5s, show "No internet — Retry" button at bottom; logo
stays
• Backend returns 503: auto-retry 3 times with 2s/4s/8s backoff
• All retries fail: route to S-COMMON-MAINT
